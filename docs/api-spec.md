# MeritTrack — API Specification

## 1. API Grouping Strategy

All endpoints are prefixed with `/api/v1/`. Groups:

| Prefix | Domain |
|---|---|
| `/api/v1/auth` | Authentication, token refresh, device registration |
| `/api/v1/idp` | Internal identity-provider token issuance and JWKS |
| `/api/v1/candidates` | Candidate profiles, exam scores, transfer preferences, document upload |
| `/api/v1/documents` | Document review decisions, document download |
| `/api/v1/services` | Service catalog (list available service items) |
| `/api/v1/orders` | Order creation, state transitions, payment, milestones, vouchers, bargaining, refunds, after-sales |
| `/api/v1/attendance` | Attendance anomaly flagging, exception creation, proof upload, routing, adjudication |
| `/api/v1/queue` | Staff queues: pending documents, payments, orders, exceptions, after-sales |
| `/api/v1/admin` | Feature flags, cohort routing, audit log, exports, observability, forecasting, bootstrap config |
| `/api/v1/internal` | Health check (unauthenticated liveness probe) and Prometheus-style metrics (admin-gated; intended for local privileged scraping — see §7.11) |

## 3. Authentication, Refresh, and Request-Signature Headers

### 3.1 Standard Request Headers

| Header | Required | Description |
|---|---|---|
| `Authorization` | Yes (authenticated) | `Bearer <access_token>` |
| `X-Request-ID` | Optional | Forwarded or generated by `TraceIdMiddleware`; echoed in response |
| `X-Nonce` | Yes (signed) | Unique value (`n-<hex>`); server rejects replays within 5-minute window |
| `X-Timestamp` | Yes (signed) | ISO 8601 UTC (`YYYY-MM-DDTHH:MM:SSZ`); rejected if outside ±30 seconds of server time |
| `X-Request-Signature` | Yes (signed) | Base64 ECDSA P-256 signature over canonical form: `METHOD\nPATH\nX-Timestamp\nX-Nonce\nX-Device-ID\nsha256_hex(body)\n` |
| `X-Device-ID` | Yes (signed) | Registered device identifier |
| `Idempotency-Key` | Optional (mutations) | 8–128 char token; body hash must match prior call for cached replay |

### 3.2 Signed-Route Inventory

Routes where the server requires canonical `X-Request-Signature` headers. Signing is enforced via the `require_signed_request` FastAPI dependency declared explicitly on each route — it is **not** middleware path-matching.

Auth / enrollment / documents:

- `POST /api/v1/auth/password/change`
- `POST /api/v1/auth/device/{device_id}/rotate`
- `POST /api/v1/candidates/{candidate_id}/documents/upload`

Order lifecycle (create + bargaining):

- `POST /api/v1/orders`
- `POST /api/v1/orders/{order_id}/bargaining/offer`

Financial / workflow mutations (added under audit-5 B2 so that every state-changing operation on a paid order is device-bound + anti-replay):

- `POST /api/v1/orders/{order_id}/payment/proof`
- `POST /api/v1/orders/{order_id}/payment/confirm`
- `POST /api/v1/orders/{order_id}/voucher`
- `POST /api/v1/orders/{order_id}/milestones`
- `POST /api/v1/orders/{order_id}/refund`
- `POST /api/v1/orders/{order_id}/refund/process`
- `POST /api/v1/orders/{order_id}/after-sales`
- `POST /api/v1/orders/{order_id}/after-sales/{request_id}/resolve`

Attendance exceptions:

- `POST /api/v1/attendance/exceptions`
- `POST /api/v1/attendance/exceptions/{exception_id}/proof`
- `POST /api/v1/attendance/exceptions/{exception_id}/review`

Order lifecycle — state transitions (added under audit-1 remediation so every order-state POST is device-bound + anti-replay; closes the residual coverage gap left after audit-5 B2 by adding signing to the cancel/confirm-receipt/advance path and the reviewer-driven bargaining acceptance path):

- `POST /api/v1/orders/{order_id}/cancel`
- `POST /api/v1/orders/{order_id}/confirm-receipt`
- `POST /api/v1/orders/{order_id}/advance`
- `POST /api/v1/orders/{order_id}/bargaining/accept`
- `POST /api/v1/orders/{order_id}/bargaining/counter`
- `POST /api/v1/orders/{order_id}/bargaining/accept-counter`

Device activation (`POST /api/v1/auth/device/activate`) is protected by a **challenge-signature** flow rather than canonical signed-request headers: the client signs the enrollment nonce returned by `/auth/device/challenge` with the device-bound private key, and the backend verifies that signature in the request body before registering the public key. This bootstrap flow runs before any `device_id` exists, so canonical `X-Device-ID` signing cannot apply. Activation is therefore intentionally absent from this list.

The `signature_required_paths` setting in `config.py` is informational only — it is used by the frontend's `SIGNED_PATHS` list in `services/http.ts` to know which outbound requests must include canonical signing headers. Backend enforcement is always via `Depends(require_signed_request)`.

### 3.3 Token Endpoints

- `POST /api/v1/auth/login` — body: `{ "username", "password", "nonce", "timestamp" }`. Response: `SuccessResponse<LoginResponse>` (200) or `AUTH_REQUIRED` (401), `VALIDATION_ERROR` (422), `AUTH_THROTTLED` (429).
- `POST /api/v1/auth/refresh` — body: `{ "refresh_token" }`. Returns rotated pair. Reuse of a consumed refresh token invalidates the entire family and subsequent attempts return `AUTH_REQUIRED`.
- `POST /api/v1/auth/logout` — body: `{ "refresh_token" }`. Invalidates the entire family regardless of which member was presented.
- `GET /api/v1/auth/me` — returns `MeResponse` = `{ user: UserRead, cohort_config?: object, device_id?: string, candidate_id?: uuid }`. `candidate_id` is the resolved `CandidateProfile.id` for callers with `role=candidate` (distinct from `user.id`, which is the auth user PK); `null` for non-candidates and for candidates whose profile row has not been created yet. Frontend views that scope data to a candidate (profile, documents, exam scores, transfer preferences, orders) MUST send `candidate_id`, not `user.id`.
- `POST /api/v1/auth/password/change` — auth + signed. Enforces 12-char new password policy and Argon2id rehash; emits `AUDIT_PASSWORD_CHANGED`.

### 3.4 Device Enrollment

Canonical browser flow is **challenge → activate** (two-step). `activate` is atomic: it verifies the challenge signature, binds the supplied public key to the caller's user, and returns the assigned `device_id` in a single transaction. The standalone `register` endpoint is reserved for service-managed direct registration (e.g., CLI tools / test devices) and is **not** part of the browser path.

1. `POST /api/v1/auth/device/challenge` (auth) — body: `DeviceChallengeRequest` = `{ device_fingerprint: string(8..128), public_key_pem: string, label?: string }`. Returns `{ challenge_id, nonce, expires_at }` (5-minute validity). The server binds `{challenge_id → device_fingerprint, user_id}` so a later activate from a different fingerprint is rejected.
2. `POST /api/v1/auth/device/activate` — body: `DeviceActivateRequest` = `{ challenge_id, device_fingerprint, public_key_pem, signature, label? }`. `signature` is ECDSA P-256 over the **raw `nonce` UTF-8 bytes** (it is NOT the request-canonical `METHOD\nPATH\nTS\nNONCE\nDEVICE_ID\nSHA256(body)\n` form used for in-session signed requests — enrollment pre-dates the device-ID, so there is nothing to key the canonical form against). Returns `DeviceRegisterResponse` = `{ device_id, device_fingerprint, registered_at }`; the response device_id is the identifier subsequent signed requests must send in `X-Device-ID`.
3. `POST /api/v1/auth/device/register` — body: `DeviceRegisterRequest` = `{ device_fingerprint, public_key_pem, label? }`. **Service-managed direct registration only**; the standard browser enrollment path is challenge → activate, no separate register call. Returns `DeviceRegisterResponse`.
4. `POST /api/v1/auth/device/{device_id}/rotate` — signed. Issues new keypair-backed registration and revokes the prior one in the same transaction.
5. `DELETE /api/v1/auth/device/{device_id}` — revokes (soft-delete): `revoked_at` is set and subsequent signed requests from that device receive `DEVICE_NOT_FOUND`.

### 3.5 Internal IdP

- `POST /api/v1/idp/token` — body: `{ "grant_type": "client_credentials", "client_id", "client_secret", "scope" }`. Unknown client → `AUTH_REQUIRED`; valid → JWT signed with the same RS256 key and `aud="internal"`.
- `GET /api/v1/idp/jwks` — RFC 7517 JWKS document containing only public components (`n`, `e`, `kty=RSA`, `alg=RS256`, `kid`, `use=sig`).

## 4. Error Model

### 4.1 Error Codes

| Code | HTTP Status | Description |
|---|---|---|
| `VALIDATION_ERROR` | 422 | Request body/query/path failed schema validation |
| `AUTH_REQUIRED` | 401 | Missing or invalid access token |
| `TOKEN_EXPIRED` | 401 | Access token past its 15-minute lifetime |
| `FORBIDDEN` | 403 | Authenticated but insufficient role/ownership |
| `NOT_FOUND` | 404 | Resource does not exist or is not visible to this role |
| `CONFLICT` | 409 | State conflict (duplicate nonce, order already in terminal state, etc.) |
| `NONCE_REPLAY` | 409 | Nonce seen within the 5-minute window |
| `SIGNATURE_INVALID` | 400 | Request signature verification failed |
| `CLOCK_SKEW` | 400 | Timestamp outside ±30-second tolerance |
| `BARGAINING_LIMIT` | 422 | Candidate has submitted maximum allowed offers |
| `BARGAINING_WINDOW_EXPIRED` | 422 | 48-hour bargaining window has elapsed |
| `AFTER_SALES_WINDOW_EXPIRED` | 422 | 14-day after-sales window has elapsed |
| `FILE_TOO_LARGE` | 422 | File exceeds 25 MB limit |
| `FILE_TYPE_NOT_ALLOWED` | 422 | File MIME type not in PDF/JPG/PNG |
| `ORDER_TRANSITION_INVALID` | 422 | Requested state transition is not permitted |
| `AUTH_THROTTLED` | 429 | Too many recent login failures for this username; retry after lockout window |
| `DEVICE_NOT_FOUND` | 400 | Signed request referenced an unknown or revoked device |
| `IDEMPOTENCY_CONFLICT` | 409 | Same `Idempotency-Key` replayed with a different body hash |
| `INTERNAL_ERROR` | 500 | Unexpected server error (details not exposed to client) |

### 4.2 Validation Error Detail Shape
```json
{
  "code": "VALIDATION_ERROR",
  "message": "Request validation failed",
  "details": [
    { "field": "password", "message": "Must be at least 12 characters" },
    { "field": "file", "message": "File size exceeds 25 MB limit" }
  ]
}
```

## 5. Idempotency-Key Conventions

- Mutating endpoints that create resources accept an optional `Idempotency-Key` header (UUID, 8–128 chars).
- If the same key is replayed within 24 hours **with the same request body hash**, the server returns the original response without re-executing the operation (same body, same status).
- If the same key is replayed with a **different body hash**, the server returns `IDEMPOTENCY_CONFLICT` (409) — the client should pick a new key.
- Idempotency records stored in `idempotency_keys` table with `(key, method, path)` uniqueness and `expires_at = created_at + 24h`. A future worker will purge expired rows.
- Applicable to: `POST /api/v1/orders` (create order). Other creation endpoints do not currently enforce idempotency.
- **Enforcement point:** the contract is enforced at the route layer via `IdempotencyStore.fetch(...)` in `src/api/routes/orders.py` **before** `OrderService.create_order` runs. On replay-with-matching-body the cached envelope is returned verbatim (same status, same JSON bytes) via `JSONResponse`. On replay-with-divergent-body the store raises `IdempotencyConflictError`, which the global exception handler renders as `409 IDEMPOTENCY_CONFLICT`. The `orders.idempotency_key` column is retained for auditability (records which token created the row) but no longer short-circuits inside the service — any request that reaches the service layer is a verified new-or-fresh call.

### 5.1 Download role gate

- All document download routes (introduced in Prompt 4) are gated by `rbac.DOWNLOAD_APPROVED_ROLES = {reviewer, admin}`. Candidates may download their own submissions; other candidates receive `FORBIDDEN` (403).

## 6. Upload Contract

### 6.1 Document Upload
```
POST /api/v1/candidates/{candidate_id}/documents/upload
Content-Type: multipart/form-data

Path parameter:
  candidate_id: UUID — the candidate profile ID (must match authenticated candidate)

Form fields:
  file: binary (PDF/JPG/PNG, max 25 MB)
  requirement_code: string (optional) — links upload to a checklist requirement

Response 201:
{
  "data": {
    "document_id": "uuid",
    "version_number": 1,
    "original_filename": "original_name.pdf",
    "content_type": "application/pdf",
    "size_bytes": 1048576,
    "sha256_hash": "abc123...",
    "status": "pending_review",
    "uploaded_at": "2025-01-01T12:00:00Z"
  }
}
```

### 6.2 Document Download
```
GET /api/v1/documents/{document_id}/download

Response headers:
  Content-Type: application/pdf (or image/jpeg, image/png)
  X-File-Hash: sha256:<hash_of_served_content>
  Content-Disposition: attachment; filename="watermarked_<original>.pdf"

PDF files: watermarked with requesting username + ISO timestamp before delivery
Non-PDF: served directly with hash header
```

## 9. Sensitive Field Masking Rules

The following fields are masked in API responses unless the requesting role has explicit access:

| Field | Default Mask | Full Access Roles |
|---|---|---|
| `ssn` | `***-**-{last4}` | ADMN |
| `date_of_birth` | `YYYY-**-**` | REVW, ADMN |
| `contact_phone` | `***-***-{last4}` | REVW, ADMN |
| `contact_email` | `{first}***@***.{tld}` | REVW, ADMN |
| `bank_account_ref` | `****{last4}` | ADMN |

Masking is applied in the API serialization layer, not the database layer. The database always stores encrypted ciphertext.

## 10. Concrete Payload Contracts (Prompt 2)

### 10.1 Auth

**POST /api/v1/auth/login** — Request:
```json
{
  "username": "string (1–150 chars)",
  "password": "string (min 12 chars)",
  "device_fingerprint": "string|null (max 128)",
  "nonce": "string (UUID, max 64)",
  "timestamp": "ISO 8601 UTC string"
}
```
Response 200: `SuccessResponse<TokenResponse>` — `access_token`, `refresh_token`, `token_type`, `expires_in`, `user_id`, `role`, `cohort_config`.

**POST /api/v1/auth/refresh** — Request: `{ "refresh_token": "...", "nonce": "...", "timestamp": "..." }`. Response 200: `SuccessResponse<RefreshResponse>`.

**POST /api/v1/auth/device/register** — Request: `{ "device_fingerprint": "...", "public_key_pem": "...", "label": "..." }`. Response 201: `SuccessResponse<DeviceRegisterResponse>`.

**POST /api/v1/auth/password/change** — Request: `{ "current_password": "...", "new_password": "string (min 12 chars)", "nonce": "...", "timestamp": "..." }`.

### 10.2 Candidates

**GET /api/v1/candidates/{id}** — Response: `SuccessResponse<CandidateProfileRead>`. Sensitive fields masked per role (see §9). Encrypted fields returned as masked display strings, never raw ciphertext.

**PUT /api/v1/candidates/{id}** — Request body: `CandidateProfileUpdate` (`preferred_name`, `application_year`, `program_code`, `notes`). Only CAND (own profile) and REVW/ADMN.

**POST /api/v1/candidates/{id}/scores** — Request: `ExamScoreCreate` (`subject_code`, `subject_name`, `score`, `max_score`, `exam_date`). PRCT/REVW/ADMN only.

**POST /api/v1/candidates/{id}/transfer-preferences** — Request: `TransferPreferenceCreate` (`institution_name`, `program_code`, `priority_rank ≥ 1`, `notes`).

### 10.3 Documents

**POST /api/v1/candidates/{candidate_id}/documents/upload** — `multipart/form-data`: `file` (PDF/JPEG/PNG, ≤ 25 MB), `requirement_code` (optional string). Candidate must match authenticated user's profile. Response 201: `SuccessResponse<DocumentUploadResponse>` with `document_id`, `version_number`, `original_filename`, `sha256_hash`, `status`.
Error codes: `FILE_TOO_LARGE`, `FILE_TYPE_NOT_ALLOWED`, `FORBIDDEN`.

**GET /api/v1/documents/{id}/download** — Returns binary file. PDF: watermarked with reviewer identity + timestamp. Headers: `X-File-Hash: sha256:<hash>`, `Content-Disposition: attachment`. Roles: REVW/ADMN only. Candidates can see their own document metadata via the list endpoint but cannot download the file; downloads are watermarked with reviewer identity.

**POST /api/v1/documents/{id}/review** — Request: `DocumentReviewCreate` (`version_id`, `status`, `resubmission_reason` required when `needs_resubmission`, `reviewer_notes`). REVW only. Response 200: `SuccessResponse<DocumentReviewRead>`.
Error codes: `RESUBMISSION_REASON_REQUIRED`.

### 10.4 Orders

**POST /api/v1/orders** — Request: `OrderCreate` (`item_id`, `pricing_mode`, `idempotency_key`). Response 201: `SuccessResponse<OrderRead>`. If item is capacity-limited, a slot is reserved atomically. `auto_cancel_at` is set to `now + 30 minutes`.
Error codes: `ITEM_UNAVAILABLE` (no slots).

**GET /api/v1/orders/{id}** — Response: `SuccessResponse<OrderRead>` with full `events` array ordered by `sequence_number`. Timestamps are UTC/ISO in API; frontend renders 12-hour format.

**POST /api/v1/orders/{id}/payment/confirm** — Request: `PaymentConfirmRequest` (`amount`, `payment_method`, `reference_number`, `notes`). REVW/ADMN. Transitions `pending_payment → pending_fulfillment`.
Error codes: `INVALID_TRANSITION`, `ORDER_NOT_FOUND`.

**POST /api/v1/orders/{id}/refund** — Request: `RefundCreate` (`amount`, `reason`). REVW/ADMN. Transitions `completed → refund_in_progress`. If `rollback_on_refund` flag is true and item is capacity-limited, slot is restored atomically.

### 10.5 Bargaining

**POST /api/v1/orders/{id}/bargaining/offer** — Request: `OfferCreate` (`amount > 0`, `nonce`, `timestamp`). CAND only. Signed. Max 3 offers within 48-hour window.
Error codes: `BARGAINING_OFFERS_EXHAUSTED`, `BARGAINING_WINDOW_CLOSED`, `BARGAINING_DISABLED`.

**POST /api/v1/orders/{id}/bargaining/counter** — Request: `CounterCreate` (`counter_amount > 0`, `notes`). REVW only. Max 1 counter per thread.
Error codes: `BARGAINING_COUNTER_ALREADY_MADE`.

**POST /api/v1/orders/{id}/bargaining/accept-counter** — Request: `CounterAcceptRequest` (`nonce`, `timestamp`). CAND only. Transitions order to `pending_payment` at counter amount.

### 10.6 After-Sales

**POST /api/v1/orders/{id}/after-sales** — Request: `{ "request_type": "...", "description": "..." }`. CAND only. Backend validates order is `completed` and `now <= completed_at + 14 days`.
Error codes: `AFTER_SALES_WINDOW_EXPIRED`, `ORDER_NOT_COMPLETED`.

### 10.7 Attendance Exceptions

**POST /api/v1/attendance/exceptions/{id}/review** — Request: `ReviewDecisionCreate` (`decision`: approve|reject|escalate, `notes`). PRCT for initial stage, REVW for final stage. Admin can perform both.
Error codes: `UNAUTHORIZED_STAGE`, `INVALID_DECISION_FOR_STAGE`.

**GET /api/v1/attendance/exceptions/{id}** — Response: `SuccessResponse<ExceptionRead>` with full `review_steps` array ordered by `step_order`. Each step is immutable after creation.

### 10.8 Config

**GET /api/v1/config/bootstrap** — Response: `SuccessResponse<BootstrapConfigResponse>` with `feature_flags`, `flag_overrides` (from cohort), `resolved_flags` (merged), and a `signature` (HMAC of the payload). Frontend must verify signature before trusting config.

**PUT /api/v1/config/flags/{key}** — Request: `FeatureFlagUpdate` (`value`, `change_reason`). ADMN only. Change recorded in `feature_flag_history`.

**After-sales window enforcement:**
```
Error 409: { "error": { "code": "BUSINESS_RULE_VIOLATION", "message": "After-sales window closed at ..." } }
```
After-sales resolve is independent of refund — resolving a complaint does not trigger a refund automatically.

**Parent-child path binding (resolve):** the `POST /orders/{order_id}/after-sales/{request_id}/resolve` handler verifies that the after-sales record identified by `{request_id}` actually belongs to `{order_id}`. A mismatched pair (e.g., request created under order A but addressed via order B's URL) returns `404 NOT_FOUND` and does not mutate state. Enforcement lives in `AfterSalesService.resolve` (`src/services/after_sales_service.py`).

**Resolve request body:** `AfterSalesResolveRequest` — JSON `{ "resolution_notes": string (1..5000) }`. Sent as a JSON body (not query params). Signed per §5.1.
